# Architecture

> Forge agents read this before making architectural decisions.

## Module Map

- `skills/`        — 21 skill definitions (SKILL.md each), organized by workflow phase
- `agents/`        — 5 specialist agent definitions (architect, implementer, qa-engineer, code-reviewer, security-reviewer)
- `hooks/`         — Hook scripts and registry (hooks.json) for Claude Code/Cursor lifecycle events
- `bin/`           — CLI tools: classify-risk, forge-state, forge-evidence, forge-memory, forge-pack, storage backends
- `policies/`      — Risk classification rules (default.yaml with glob to tier mapping)
- `adapters/`      — Platform-specific adapters (generated by syncing-forge)
- `workflows/`     — Workflow definitions
- `packs/`         — Installed pack bundles
- `templates/`     — Blank templates for project adoption (architecture.md, conventions.md, ADR template)
- `.forge/shared/` — This directory; team knowledge (architecture, conventions, decisions)
- `.forge/local/`  — Machine-local state (gitignored): SQLite DB, memory, evidence, audit, cache
- `docs/`          — Design docs, plans, ideation, platform-specific install guides
- `tests/`         — Test suite: skill tests, state tests, risk tests, hook tests, pack tests, integration tests, pressure tests
- `.claude-plugin/` — Claude Code plugin manifest
- `.cursor-plugin/` — Cursor plugin manifest
- `.codex/`        — Codex installation docs
- `.opencode/`     — OpenCode installation docs

## Key Patterns

- **Skill-driven workflow:** All behavior is defined in skill SKILL.md files. Skills are prompt-only — no compiled code, no API contracts. The LLM reads and follows them.
- **Intent-based routing:** `forge-routing` detects user intent and dispatches to the appropriate skill pipeline. Skills chain to each other via explicit handoff instructions.
- **Risk-scaled ceremony:** `classify-risk` reads `policies/` and returns a tier (minimal/standard/elevated/critical). The tier determines required artifacts, worktree usage, review depth, and team composition.
- **Evidence-gated completion:** `forge-evidence` records structured evidence (command output, citations, diffs). `verification-before-completion` requires evidence before allowing completion claims.
- **State persistence:** `forge-state` provides a key-value store (SQLite or JSON backend) in `.forge/local/`. State survives across sessions within the same working directory.
- **Hook lifecycle:** Hooks fire on Claude Code/Cursor events (SessionStart, TaskCompleted, PreToolUse). They initialize state, enforce gates, and run pre-commit checks.

## Dependency Graph

```
Skills (prompt-only, read by LLM)
  └── reference → bin/ tools (shell scripts)
  └── reference → policies/ (YAML rules)
  └── reference → .forge/shared/ (team knowledge)
  └── chain to → other skills (via handoff instructions)

Hooks (shell scripts, fired by host platform)
  └── call → bin/ tools
  └── read → .forge/local/ state

bin/ tools (shell scripts)
  └── read/write → .forge/local/ (SQLite or JSON)
  └── read → policies/ (YAML)
```

Skills must not import or depend on each other at a code level. They chain via LLM instruction only. `bin/` tools are self-contained shell scripts with no external dependencies beyond bash, sqlite3 (optional), and standard Unix tools.

## External Dependencies

- **Host platform:** Claude Code, Cursor, Codex, OpenCode, or Gemini CLI (provides the LLM runtime)
- **bash:** Required for all `bin/` tools and hooks
- **sqlite3:** Optional; used by the SQLite storage backend. Falls back to JSON if unavailable.
- **git:** Required for worktree operations and pre-commit hooks
- **No package manager, no build step, no compiled code.** Forge is pure shell scripts + markdown skill definitions.
